Модель **OSI** (или **O**pen **S**ystems **I**nterconnection Model) — это важнейшая модель, используемая в сетевых технологиях. Эта критически важная модель представляет собой структуру, определяющую, как все сетевые устройства будут отправлять, получать и интерпретировать данные.

Модель OSI состоит из семи уровней, которые показаны на схеме ниже. Каждый уровень выполняет свои функции и расположен в порядке от уровня 7 к уровню 1.

- 7. Application
- 6. Presentation
- 5. Seesion
- 4. Transport
- 3. Network
- 2. Data link
- 1. Physical

Процесс добавления информации - инкапсуляция (encapsulation)

# **Первый уровень - physical**

Этот уровень является одним из самых простых для понимания. Проще говоря, этот уровень относится к физическим компонентам оборудования, используемого в сетях, и является самым нижним из доступных уровней. Устройства используют электрические сигналы для передачи данных друг другу в двоичной системе счисления (1 и 0).

# **Второй уровень - Data link**

Уровень канала передачи данных отвечает за физическую адресацию при передаче. Он получает пакет от сетевого уровня (включая IP-адрес удалённого компьютера) и добавляет физический **MAC** (Media Access Control) адрес принимающей конечной точки. Внутри каждого компьютера с поддержкой сети находится **N**etwork **I**nterface Card (**NIC**), которая имеет уникальный MAC-адрес для идентификации.

MAC-адреса устанавливаются производителем и буквально «записываются» в карту. Их нельзя изменить, но можно подделать. При отправке информации по сети именно физический адрес используется для определения того, куда именно отправлять информацию.

Кроме того, задача канального уровня — представить данные в формате, подходящем для передачи.

# **Третий уровень - Network**

^d2654a

На третьем уровне модели OSI (сетевом уровне) происходит волшебство маршрутизации и повторной сборки данных (из этих небольших фрагментов в более крупный фрагмент). Во-первых, маршрутизация просто определяет наиболее оптимальный путь для отправки этих фрагментов данных.

Хотя некоторые протоколы этого уровня точно определяют «оптимальный» путь, по которому данные должны пройти, чтобы попасть на устройство, на этом этапе работы сетевого модуля мы должны знать только об их существовании. Если вкратце, то к таким протоколам относятся **OSPF** (**O**pen **S**hortest **P**ath **F**irst) и **RIP** (**R**outing **I**nformation **P**rotocol). Выбор маршрута зависит от следующих факторов:

- Какой путь самый короткий? То есть на нём меньше всего устройств, через которые должен пройти пакет данных.
- Какой маршрут наиболее надёжен? То есть случались ли потери пакетов на этом маршруте раньше?
- Какой канал обеспечивает более быстрое физическое соединение? То есть какой канал использует медное соединение (более медленное) или оптоволоконное (значительно более быстрое)?

На этом уровне всё взаимодействует через IP-адреса, такие как 192.168.1.100. Такие устройства, как **маршрутизаторы, способные доставлять пакеты данных по IP-адресам, называются устройствами третьего уровня**, поскольку они могут работать на третьем уровне модели OSI.

# **Четвертый уровень - Transport**
	Четвёртый уровень модели OSI играет важную роль в передаче данных по сети, но разобраться в нём может быть непросто. При передаче данных между устройствами используется один из двух протоколов, выбор которых зависит от нескольких факторов:
	
## **TCP**
Давайте начнём с **TCP - Transmission Control Protocol**. Как следует из названия, этот протокол разработан с учётом требований к надёжности и гарантиям. Этот протокол обеспечивает постоянное соединение между двумя устройствами на время, необходимое для отправки и получения данных.
Кроме того, TCP включает в свою структуру проверку на наличие ошибок. Проверка на наличие ошибок позволяет TCP гарантировать, что данные, отправленные небольшими фрагментами на сеансовом уровне (уровень 5), были получены и собраны в том же порядке.

Давайте обобщим преимущества и недостатки TCP в таблице ниже:

| **Преимущества TCP**                                                            | **Недостатки TCP**                                                                                                                                        |
| :------------------------------------------------------------------------------ | :-------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Гарантирует точность данных                                                     | Требуется надёжное соединение между двумя устройствами. Если не получен один небольшой фрагмент данных, то весь фрагмент данных не может быть использован |
| Способен синхронизировать два устройства, чтобы предотвратить перегрузку данных | Медленное соединение может стать узким местом для другого устройства, поскольку соединение будет постоянно зарезервировано на принимающем компьютере      |
| Выполняет гораздо больше процессов для обеспечения надёжности.                  | TCP работает значительно медленнее, чем UDP, поскольку устройствам, использующим этот протокол, приходится выполнять больше работы                        |

TCP используется в таких ситуациях, как обмен файлами, просмотр веб-страниц или отправка электронных писем. Это связано с тем, что для этих сервисов требуется точность и полнота данных (нехорошо, если файл будет неполным!).

## **UDP**

Теперь давайте перейдём к протоколу **U**ser **D**atagram **P**rotocol (или сокращённо **UDP**). Этот протокол далеко не так продвинут, как его брат — протокол TCP. Он не может похвастаться множеством функций, предлагаемых TCP, таких как проверка ошибок и надёжность. На самом деле любые данные, отправляемые через UDP, доходят до компьютера независимо от того, получены они или нет. Между двумя устройствами нет синхронизации или гарантии. Остается только надеяться на лучшее и держать кулачки.

Хотя это и звучит не слишком привлекательно, у такого подхода есть свои преимущества, которые мы представим в таблице ниже:

| Преимущества UDP                                                                                                                                 | Недостатки UDP                                                                     |
| ------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------- |
| UDP работает намного быстрее, чем TCP.                                                                                                           | UDP не заботится о том, получены ли данные.                                        |
| UDP оставляет за прикладным уровнем (пользовательским программным обеспечением) право решать, можно ли контролировать скорость отправки пакетов. | В этом смысле он довольно гибок для разработчиков программного обеспечения.        |
| UDP не резервирует постоянное соединение на устройстве, как это делает TCP.                                                                      | Это означает, что нестабильное соединение доставляет пользователю массу неудобств. |
UDP полезен в ситуациях, когда передаются небольшие фрагменты данных. Например, протоколы, используемые для обнаружения устройств (_ARP_ и _DHCP_  или для передачи больших файлов, таких как потоковое видео (где допустимо, если какая-то часть видео будет пиксельной. Пиксели — это просто потерянные фрагменты данных!)

# **Уровень пятый - Session**
После того как данные будут правильно переведены или отформатированы на уровне представления (уровень 6), уровень сеанса (уровень 5) начнёт устанавливать и поддерживать соединение с другим компьютером, для которого предназначены данные. После установления соединения создаётся сеанс. Пока соединение активно, активен и сеанс.

Уровень сеанса также отвечает за закрытие соединения, если оно не использовалось в течение некоторого времени или было потеряно. Кроме того, сеанс _может_ содержать «контрольные точки», при потере которых отправляются только самые новые данные, что позволяет экономить трафик.

Стоит отметить, что сеансы уникальны, то есть данные не могут передаваться между разными сеансами, а передаются только в рамках одного сеанса.

# **Уровень шестой - Presentation**
Уровень 6 модели OSI — это уровень, на котором начинается стандартизация. Поскольку разработчики программного обеспечения могут создавать любое ПО, например почтовый клиент, по-разному, данные должны обрабатываться одинаково, независимо от того, как работает программное обеспечение.

Этот уровень выступает в роли переводчика данных между прикладным уровнем (уровень 7) и другими уровнями. Получающий компьютер также будет понимать данные, отправленные на компьютер в одном формате, но предназначенные для другого формата. Например, когда вы отправляете электронное письмо, у другого пользователя может быть другой почтовый клиент, но содержимое письма должно отображаться одинаково.

На этом уровне реализуются такие функции безопасности, как шифрование данных (например, HTTPS при посещении защищенного сайта).
Основная функция слоя - перевод данных между 7 уровнем и остальными (Translator)

# **Уровень седьмой - Application**
Уровень приложений в модели OSI — это тот уровень, с которым вы знакомы лучше всего. Это связано с тем, что на уровне приложений действуют протоколы и правила, определяющие, как пользователь должен взаимодействовать с отправляемыми или получаемыми данными.

Повседневные приложения, такие как почтовые клиенты, браузеры или программы для просмотра файловых серверов, например FileZilla, предоставляют удобный Graphical Users Interface (**GUI**) для взаимодействия пользователей с отправляемыми или получаемыми данными. К другим протоколам относится **DNS** (Domain Name System), с помощью которой адреса веб-сайтов преобразуются в IP-адреса.
